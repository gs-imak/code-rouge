## How to use this file

Each chantier is broken into ordered tasks with **acceptance criteria** that
must be satisfied before the chantier is considered done. Work them in order.
Do not start chantier N+1 until chantier N's acceptance criteria are all met.

When picking up a task:
1. Read the chantier's "Goal" and the task's "AC" (acceptance criteria).
2. Outline the implementation before coding (notes, branch plan, etc.).
3. Implement, verify each AC empirically (run the command, observe the result).
4. Tick the AC mentally; do not edit this file to mark progress (use Git
   commits instead).

## Chantier 03 — Serveur local NUC

**Goal:** A Node.js server that runs on the NUC (or on Georges's dev machine
during M1), exposes WebSocket endpoints for the three apps, has a SQLite
database for session state, and can be installed via a script.

**Day:** J3 — Thu 30 April

### Tasks

#### 3.1 — Server scaffold

In `apps/server-nuc`:

- Init as ESM TypeScript project (`"type": "module"`).
- Deps: `express`, `ws`, `better-sqlite3`, `zod`, `pino`, `pino-pretty` (dev only).
- Dev deps: `tsx` (dev runner), `@types/express`, `@types/ws`, `@types/better-sqlite3`.
- Scripts: `dev` (tsx watch), `build` (tsc), `start` (node dist).
- `src/index.ts` boots Express on `:8080`, attaches a `ws.Server` on `/ws`,
  responds 200 OK on `GET /health`.

**AC:**
- `pnpm dev --filter @code-rouge/server-nuc` starts the server.
- `curl http://localhost:8080/health` returns 200.
- A `wscat -c ws://localhost:8080/ws` connection succeeds.

#### 3.2 — WebSocket message protocol

In `packages/shared-types/src/messages.ts`, define **Zod schemas** for:

- `HelloMessage` — app announces itself: `{ type: 'hello', app: 'attaque-de-bots' | 'assaut' | 'debriefing', deviceId: string, teamId: number | null }`
- `StateUpdateMessage` — app reports state: `{ type: 'state', app, deviceId, teamId, step: string, score: number, timestamp: number }`
- `LogPushMessage` — `{ type: 'log', app, deviceId, teamId, events: LogEvent[] }`
- `ServerCommandMessage` — `{ type: 'cmd', cmd: 'ping' | 'reset' | 'sync' }`

Export `parseMessage(raw: string)` that returns a discriminated union or throws
a typed `MessageParseError`.

**AC:**
- `pnpm typecheck` passes for `shared-types`.
- A unit test in `messages.test.ts` covers (a) valid hello round-trip, (b) malformed JSON throws, (c) unknown type throws.

#### 3.3 — SQLite persistence

In the server:

- Create `data/coderouge.sqlite` on first boot. Path configurable via `DATABASE_PATH` env var.
- Migrations: a single `migrations/001_init.sql` with tables `sessions`, `team_state`, `event_log`.
- On every `StateUpdateMessage`, upsert into `team_state` keyed by `(sessionId, teamId, app)`.
- On every `LogPushMessage`, append to `event_log`.

**AC:**
- Send a state update via `wscat`, observe the row in SQLite (`sqlite3 data/coderouge.sqlite "SELECT * FROM team_state"`).
- Restart the server, run the same SELECT — the row is still there.

#### 3.4 — Deploy script

Create `tools/scripts/install-nuc.sh`:

- Installs Node 22 (via NodeSource or nvm).
- Clones the repo, runs `pnpm install --prod`, builds the server.
- Sets up a `systemd` unit (`code-rouge-server.service`) that runs the server on boot.
- Logs to `/var/log/code-rouge/`.

Document in `apps/server-nuc/README.md` how to run it on a fresh Ubuntu LTS NUC.

**AC:**
- Script lints clean with `shellcheck`.
- Dry-run on Georges's dev machine (or in a Docker container) reaches the
  systemd-enable step without errors.

## Chantier 04 — Mode kiosque (Android + Windows)

**Goal:** All three player apps launch in unbreakable kiosk mode. A user
cannot exit using any combination of system shortcuts or hardware buttons
short of a force-power-off.

**Day:** J4 — Fri 1 May

### Tasks

#### 4.1 — React Native scaffolding (Attaque de Bots + Débriefing)

In `apps/attaque-de-bots` and `apps/debriefing`:

- Scaffold a fresh React Native app (latest stable). Use the bare TypeScript
  template, not Expo (we need native modules for Screen Pinning).
- Configure Metro for the monorepo: `metro.config.js` must add the workspace
  root to `watchFolders` and `nodeModulesPaths`.
- Each app has a single screen showing "Connexion équipe — placeholder"
  centered, in the temporary palette.

**AC:**
- `pnpm android --filter @code-rouge/attaque-de-bots` builds and installs
  on a connected device or emulator.
- App opens, shows the placeholder screen.

#### 4.2 — Screen Pinning on Android

- Add a `KioskModule` (Java/Kotlin native module) that calls
  `ActivityManager.startLockTask()` on app start.
- Wrap with a permission check; if Screen Pinning is not granted, show an
  on-screen instruction to enable it in Settings → Security → Screen Pinning.
- Disable the Back button in the root navigator (return `true` from the
  back handler).

**AC:**
- Once the app is running and pinned, **all** of these fail to exit:
  Home button, Recents button, Back button, swipe gestures, notification shade.
- Force-stop via Settings still works (we don't fight the OS-level kill).

#### 4.3 — Electron scaffolding (Assaut)

In `apps/assaut`:

- Init Electron + Vite + React TypeScript template. (electron-vite is fine.)
- Main process: `BrowserWindow` with `kiosk: true`, `fullscreen: true`,
  `frame: false`, `autoHideMenuBar: true`, `webPreferences.contextIsolation: true`.
- Renderer: a single screen "Section 13 — Saisie code autorisation — placeholder".

**AC:**
- `pnpm dev --filter @code-rouge/assaut` opens a fullscreen Electron window.
- Window cannot be minimized, resized, or closed via the title bar (there is none).

#### 4.4 — Windows shortcut blocking

- Use `globalShortcut.register` to swallow `Alt+Tab`, `Alt+F4`, `Ctrl+Esc`,
  `Super+L`, `Super+D`, `Ctrl+Shift+Esc` (Task Manager).
- For deeper hooks (Win key alone, Ctrl+Alt+Del — the latter is **not**
  blockable from user-mode and that's by design), document the limitation
  in `apps/assaut/README.md`. The proposition's "triple verrou" plans for
  this: the third lock is a Windows session policy on the deployed machine,
  configured by Nathanael's team, not by us.

**AC:**
- All listed shortcuts are no-ops while Assaut is focused.
- README documents the Ctrl+Alt+Del limitation and points to the
  client-side mitigation (session restriction).

## Chantier 05 — Persistance & reprise automatique

**Goal:** Every app resumes exactly where it stopped after a forced
restart, on all three platforms.

**Day:** J5 — Mon 4 May (morning) ; demo to Nathanael at 14h30.

### Tasks

#### 5.1 — Local state store (RN apps)

In each RN app:

- Use `@react-native-async-storage/async-storage`.
- Define a `useGameState()` hook that wraps `(state, setState)`, persists
  on every change, and rehydrates on app boot before the first render.
- Persist: `teamId`, `currentStep`, `score`, `lastSync` timestamp.

**AC:**
- Open Attaque de Bots, set teamId=7 via the placeholder admin screen,
  navigate forward two steps, force-kill the app via `adb shell am force-stop`.
- Reopen — app shows team 7 at step 2. No flash of "team selection" screen.

#### 5.2 — Local state store (Electron app)

In Assaut:

- Use `electron-store` (or just a typed JSON file in `app.getPath('userData')`).
- Same shape and behaviour as 5.1.
- On main process startup, load state synchronously and pass it to the
  renderer via the initial IPC handshake.

**AC:**
- Type a code on the placeholder screen, force-kill via Task Manager (since
  the kiosk blocked the rest), reopen — code is still in the input field.

#### 5.3 — Server-side resume on connect

When a `HelloMessage` arrives on the server WebSocket:

- Look up `(teamId, app)` in `team_state`.
- If found, send back a `StateRestoreMessage` with the last known state.
- The app, on receiving it, reconciles (server is authoritative for
  cross-app fields like teamId, app is authoritative for its own UI step).

**AC:**
- Reset Attaque de Bots's local storage. Reconnect to the server.
  App receives the team's last-known step from the server and lands on it
  without going through team selection.

#### 5.4 — Persistance demo prep

- Write a short script `tools/scripts/demo-persistence.sh` that runs the
  validation scenario from `docs/m1-plan.md#validation` (below) automatically
  where possible (`adb shell am force-stop`, server health check).
- Take screenshots of each step into `docs/demo/m1/`.
- Tag the validation commit and update the README's status line.

**AC:**
- Script runs end to end without manual intervention except the physical
  reboot of the Windows PC (which can't be scripted from inside).
- Screenshots match the 5-step demo scenario.
